Objective-C

OC特性

  1. 面向对象语言特性：封装，继承，多态；
  2. 动态类型：运行时再决定对象的类型，比如id；
  3. 动态绑定：基于动态类型，在某个实例对象被确定后，其类型便被确定了，该对象对应属性和响应消息也被确定了
  4. 动态加载：根据需求加载所需资源，最基本的适配机型。

深拷贝浅拷贝区别

  1. 浅拷贝只是复制了对象的引用地址，两个对象指向同一个内存地址，所以修改其中任意的值，另一个值都会随之变化，这就是浅拷贝（例：assign()）
  2. 深拷贝是将对象及值复制过来，两个对象修改其中任意的值另一个值不会改变，这就是深拷贝（例：JSON.parse()和JSON.stringify()，但是此方法无法复制函数类型）

线程进程

一个程序至少有一个进程，一个进程至少一个线程；

进程：一个程序的一次运行，在执行过程中拥有独立的内存单元，而多个线程共享一块内存；

线程：进程内的一个执行单元；

联系：线程是进程的基本组成单位

区别：1)调度：线程作为调度和分配的基本单位，进程作为拥有资源的基本单位；2)并发性：不仅进程之间可以并发执行，同一个进程的多个线程之间也可以并发执行；3)拥有资源：进程是拥有资源的一个独立单位，线程不拥有系统资源，但可以访问隶属进程的资源；4)系统开销：在创建或者撤销进程是，由于系统都要为之分配和回收资源，导致系统的开销明显大于创建或者撤销线程时。

UIView和CALayer区别

  1. UIView和CALayer都是UI操作的对象，都是NSObject的子类，发生在UIView上的操作本质上，也发生在对应的CALayer上
  2. UIView是CALayer用于交互的对象，UIView是UIResponder的子类，其中提供了很多CALayer没有的交互上的接口，主要负责用户触发的各种操作
  3. CALayer在图像和动画渲染上性能更好，因为UIView有冗余的交互接口，而且相比CALayer有层级之分。

iOS实现动画的方式

  1. UIView Animation:可以实现基于UIView的简单动画，它是CALayer Animation的封装，主要可以实现移动，旋转，缩放，变色等等。它还有关键帧动画和两个view转换等接口，它实现的动画无法回撤，暂停，与手势交互。
  2. CALayer Animation是更低层CALayer上的动画接口，除了可以实现UIView Animation的效果，还可以修改更多属性以实现各种复杂的动画效果，可以回撤，暂停，与手势交互
  3. UIViewPropertyAnimator是iOS10引进的处理交互动画的接口。

内存管理

OC内存管理遵循”谁创建，谁释放，谁引用，谁管理“的机制，当创建或引用一个对象时，需要向它发送alloc,copy,retain消息，当释放该对象时需发送release消息，当引用计数为0时，系统将释放该对象；

注意

  1. 代理使用assign/weak修饰
  2. block用copy修饰

SEL:类成员的方法指针，不同于C中的函数指针，SEL只是一个编号。

IMP: 函数指针，指向我们定义的函数

实现weak后，为什么对象释放后会自动为nil?

runtime对注册的类会进行布局，会将weak的对象放入一个hash表中，用weak指向的对象内存地址作为key，当该对象引用计数为0时就会调用对象dealloc方法。如weak指向的对象内存地址为a，那么以a为key去hash中搜索找到所有以a为key的weak对象全部置为nil。

循环引用

循环引用是指两个或者两个以上对象互相强引用，导致所有对象无法被释放的现象；可以用Debug Memory Graph检查，还有Runtime自动汇报；

当block本身不被self持有，而被别的对象持有，同时不产生循环强引用时；UIView的动画：UIView的某个负责动画的对象持有了block，block持有了self，但是self不持有block，就不会产生循环引用；

KVC

KVC提供了一种间接访问其属性方法或成员变量的机制，可以通过字符串来访问对应的属性方法或成员变量；

KVC原理：通过运用类型混合指针机制(isa-swizzling)来实现其内部定位查找,isa指针指向维护发布表的对象类，该发布表实际上包含了指向实现类中的指针和其他数据。

  1. 能够对对象的私有成员进行取值赋值。
  2. 对数值和结构体的属性进行的打包解包处理。

赋值过程

  1. 先找相关方法set<Key>; _set<Key>; setIs<Key>;；
  2. 若是没有相关方法，+(BOOL)accessInstanceVariablesDirectly判断是否可以直接访问成员变量；
  3. 如果判断NO,直接执行KVC的setValue:forUndefinedKey:
  4. 如果判断YES继续找相关变量_<key> _is<Key> <key> is<Key>；
  5. 方法或成员都不存在，setValue:forUndefinedKey:方法默认是抛出异常

取值过程

  1. 先找相关方法get<Key>,key
  2. 若没有相关方法，+(BOOL)accessInstanceVariabkesDirectly判断是否可以直接访问成员变量
  3. 如果是NO，直接执行KVC的valueForUndefinedKey:
  4. 如果是YES，继续找相关变量_<key>、_is<Key>、<key>、is<Key>
  5. 方法或成员都不存在，valueForUndefineKey:方法，默认是抛出异常

KVC异常处理

  1. 赋值为nil，setNilValueForKey
  2. Key值不存在，setValue:forUndefinedKey

KVO

原理:KVO是基于runtime机制实现的,当某个类的属性对象第一次被观察时，系统就会在运行期动态地创建该类的一个派生类，在这个派生类中重写基类中任何被观察属性的setter方法从而实现真正的通知机制。

  1. 如原类Person，那么就会生成一个NSKVONotifying_Person派生类；
  2. 每个类对象中都有一个isa指针指向当前类，当一个类被第一次观察，系统会将isa指针指向动态生成的派生类，从而在给被监控属性赋值时执行的是派生类的setter方法。
  3. 键值观察通知依赖于NSObject的两个方法willChangeValueForKey:和didChangevlueForKey:；在一个被观察属性发生改变前，willChangeValueForKey:调用记住旧的值，改变发生后didChangeValueForKey:调用，继而observeValueForKey:ofObject:change:context:被调用。

是否可以把比较耗时操作放在NSNotificationCenter中

  1. 如果在异步线程的通知可以执行比较耗时的操作；
  2. 如果在主线程的通知不可以执行比较耗时的操作；

比较关键字

  1. strong:表示指向并拥有该对象。其修饰的对象引用计数都会增加1。该对象只要引用计数不为0，就不会被销毁。强行置为nil可以销毁。
  2. weak:表示指向但不拥有该对象。其修饰的对象引用计数不会增加，无须手动设置，该对象会自行在内存中被销毁。
  3. assign:主要用于修饰基本数据类型，如NSInteger,CGFloat,这些数值主要存在于栈，由系统分配和释放。
  4. copy:于strong类似，不同的是strong的复制是多个指针指向同一个地址，而copy的复制是每次会在内存中复制一份对象，指针指向不同地址。copy一般用于修饰有对应可变类型的不可变对象上，如NSString,NSArray。
  5. atomic:修饰的对象会保证setter和getter的完整性，任何线程访问它都可以得到一个完整的初始化后的对象。因为要保证操作完成，所以速度比较慢。但也不是绝对的线程安全，如多个线程同时调用set和get。想要绝对安全@synchronized
  6. nonatomic:修饰的对象不保证setter和getter的完整性，所以当多线程访问它时，它可能会返回未初始化的对象，所以速度快但是线程也是不安全的。
  7. _ _weak:与weak基本相同，前者用于修饰变量，后者用于修饰属性。_ _weak主要用于防止block中的循环引用
  8. _ _block:用于修饰变量，它是引用修饰，所以它修饰的值是动态变化的，可以被重新赋值，_ _block用于修饰某些block内部将要修改的外部变量。

什么是block？它和代理的区别是什么？

block就是带有自动变量(就是局部变量)值的匿名函数。

block和代理的区别首先在于，block集中代码块，而代理分散代码块，所以block更适用于轻便，简单的回调，如网络传输。而代理适用于公共接口较多的情况，这样做也更易于解耦代码架构。另一个区别在于block运行成本高。block出栈时，需要将使用的数据从栈内存复制到堆内存，如果是对象就是加计数，使用完或block置为nil后才消除；delegate只是保存了一个对象指针，直接回调，并没有额外消耗，相对于C函数指针，只是多做了一个查表动作。

runloop

runloop基本作用

  1. 保持程序持续运行，程序一启动就会开一个主线程，主线程一开起来就会跑一个主线程对应的runloop,runloop保证主线程不会被销毁，也就保证了程序的持续运行。
  2. 处理App中的各种事件，比如触摸事件，定时器事件，Selector事件等
  3. 节省CPU资源，提高程序性能，程序运行起来时，当什么操作都没有做的时候，runloop就会告诉CPU，现在没有事情做，我休息，这是CPU就会将其资源释放出来去做其他事情，当有事情的时候runloop就会跑起来做

runloop和线程有什么关系

runloop是每一个线程一直运行的一个对象，它主要用来负责响应需要处理的各种事件和消息。每一个线程都有且仅有一个runloop与其对应，没有线程就没有runloop。

在所有线程中，只有主线程的runloop是默认启动的，main函数会设置一个NSRunLoop对象。其他线程的runloop默认是没有启动，可以通过[NSRunLoop currentRunLoop]来启动。

滑动页面上的列表时，Timer会暂停，为什么？

原因：滑动列表时，当前线程的runloop切换了model的模式导致timer暂停。

runloop中的model主要用来指定事件在runloop中的优先级。

  1. NSDefaultRunLoopMode:默认设置，一般情况下使用
  2. NSConnectionReplyMode:用于处理NSConnection相关事件
  3. NSModelPanelRunLoopMode:用于处理modal panels事件
  4. NSEventTrackingRunLoopMode:用于处理拖拽和用户交互的模式
  5. NSRunLoopCommonModes:模式合集，包括前三种；

上述情况就是滑动列表时，runloop的mode由Default模式切换到了Event Tracking三大模式，timer原来运行在Default模式中，被关闭后自然就停止了。

解决方法1：将timer加入NSRunloopCommonModes中。

[[NSRunLoop currentRunLoop] addTimer:timer forMode:NSRunLoopCommonModes]

解决方法2：将timer放入另外一个线程中，然后开启另外一个线程的runloop。

dispatch_async(dispatch_get_global_queue(0, 0), ^{
    timer = [NSTimer scheduledTimerWithTimeInterval:1 target:self selector:@selector(repeat:) userInfo:nil repeats:true];
    [[NSRunLoop currentRunLoop] run];
})

  1. mode模式的应用：利用mode互斥关系

  2. Timer加入哪个mode在滑动时是否需要继续计时，如我们如果使用scheduledTimerWithTimeInterval: 初始化Timer，则Timer只会在NSDefaultRunLoopMode下执行，如果timer需要执行的任务比较耗时，则可避免影响滑动体验。如果想要Timer在滑动时也能执行，可以使用timerWithTimeInterval:方法，并将其加入NSRunLoopCommonModes当中

  3. 线程保活（AFNetworking2.0的常驻线程、使用EarlGrey延时或保活）
  4. 卡顿监控（fps监控、CPU占用率很高、主线程RunLoop执行时间过长）。主要思路是开启一个单独子线程的RunLoop去循环检测主线程RunLoop的执行时间。卡顿标准是主线程某个RunLoop执行时间超过了1s，其检测策略是每1秒检测一次，检测从 kCFRunLoopBeforeSources 到 kCFRunLoopBeforeWaiting 花费的时间是否过长，如果检测到时间过长则Dump所有线程信息。
  5. 异步绘制（YYAsyncLayer中的YYTransaction、Texture（ASDK）中的ASAsyncTransactionGroup），利用RunLoop的回调时机及事件优先级，YYAsyncLayer与Texture实现类似，现已YYTransaction。

  6. reloadData完成时机（利用UI绘制时机与dispatch main queue的调用时机 ）。reloadData是一个异步方法，并不会等待UITableView或者UICollectionView真正刷新完毕后才执行后续代码，但有些时候我们会希望拿到reloadData调用完成的时机。这时候我们结合RunLoop，目前拿到reloadData完成时机方法有两个。

远程推送原理

  1. app向iOS系统申请远程消息推送权限
  2. iOS系统向APNS服务器请求手机端的DeviceToken,并告诉app允许推送
  3. app接收手机端的deviceToken
  4. app将收到的deviceToken传给对应服务器端
  5. 远程消息由app对应服务器产生，它先经过APNS服务器
  6. APNs服务器将远程通知推送给响应手机
  7. 根据对应deviceToken，通知会推送到指定手机

堆和栈的区别

  1. 栈区(stack)由编译器自动分配释放 ,存放方法(函数)的参数值, 局部变量的值等，栈是向低地址扩展的数据结构，是一块连续的内存的区域。即栈顶的地址和栈的最大容量是系统预先规定好的。
  2. 堆区(heap)一般由程序员分配释放, 若程序员不释放,程序结束时由OS回收，向高地址扩展的数据结构，是不连续的内存区域，从而堆获得的空间比较灵活。

启动优化

App启动时间 = main()函数(pre-main)之前的加载时间(t1)+main()函数之后的加载时间(t2);

t1 = 系统动态链接库和自身app可执行文件的加载；

t2 = application:didFinishLaunchingWithOptions:方法执行结束前这段时间

t1:

  1. 尽量使用静态库，减少动态库使用。
  2. 如果使用动态库尽量将多个dylib动态库合并成一个
  3. 减少Objective-C的类数量，例如合并或者删除，这样可以加快动态链接，重定位/绑定所耗时间减少。
  4. 删除无用的静态变量，删除无用方法。
  5. 将不必在+load中做的事情移到+initialize中
  6. 尽量不用C++虚函数，创建虚函数表有开销。
  7. 在初始化方法中不创建线程。
  8. 尽量避免对系统库使用 optional linking，如果app用到系统库在你所有支持的系统版本上都有，就设置required，因为optional会额外的检查。
  9. 不要使用__atribute__((constructor))将方法显式标记为初始化器，而是让初始化方法调用时才执行。
  10. 在初始化方法中不调用 dlopen()，dlopen()有性能和死锁的可能性

t2:

  1. 尽量不用使用Xib/storyboard。
  2. 如果使用Xib/storyboard，在当中不要存放太多视图。
  3. 对application:didFinishLaunchingWithOptions:里的任务尽量延迟加载或懒加载。
  4. 不要在NSUserDefaults 中存放太多的数据，需要反序列化一次。
  5. 避免在启动时打印过多的 log
  6. 少用 NSLog，因为每一次 NSLog 的调用都会创建一个新的 NSCalendar 实例
  7. 为了防止使用 GCD 创建过多的线程，解决方法是创建串行队列, 或者使用带有最大并发数限制的 NSOperationQueue
  8. 不要在主线程执行磁盘、网络、Lock 或者 dispatch_sync、发送消息给其他线程等操作

Swift

引用类型和值类型有什么区别？struct相比于class，优势在哪？

引用类型，如类(class)，是在堆上面进行存储和操作的；值类型，如结构体(struct)，是在栈上面进行存储操作的。堆上操作更加复杂耗时。

class

可以继承，子类可以使用父类的特性和方法；类型转换可以在运行时检查和解释一个实例类型；可以用deinit来释放资源；一个类可以被多次引用；

struct

结构较小，适用于复制操作，相比class多次引用，更加安全；无须担心内存泄露或多线程冲突问题。

Swift是面向对象还是函数式编程语言

Swift支持类的封装，继承和多态，Swift支持map,reduce,filter,flatmap这类去除中间状态的函数式方法，更加强调运算结果而不是过程，所以Swift两者都是；
